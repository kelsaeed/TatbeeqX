# 31 — Internationalization (i18n) foundation

Phase 4.5 lays the foundation for multi-language support: the layout flips correctly between LTR and RTL, date / number formatting follows the active locale, and the user can switch language from the topbar. **Translation of UI strings is not done yet** — that's a downstream task once we have actual translators.

- Lib: [`lib/core/i18n/locale_controller.dart`](../frontend/lib/core/i18n/locale_controller.dart)
- Storage: extended `TokenStorage` with free-form `readKey(key)` / `writeKey(key, value)` so the chosen locale persists in `%APPDATA%\TatbeeqX\auth.json`
- Wired into: `lib/app.dart` (MaterialApp.router) and the topbar `_LocaleSwitcher`

## What works today

- **LTR ↔ RTL flips automatically** when the user picks Arabic. The `Directionality` is set by `Localizations` based on the locale code; every Flutter widget that respects `EdgeInsetsDirectional` and `TextDirection` (most do) flips correctly.
- **Date formatting** via `intl` honours the locale.
- **Material's built-in widgets** (DatePicker, TimePicker, snackbars, dialogs) translate their own labels via `GlobalMaterialLocalizations`.
- **Locale is persisted** across runs via the new `TokenStorage` keys.

## What is NOT done (yet)

- **App-defined strings still hard-coded in English.** "Sign in", "Companies", "Approvals", etc. are literal Dart strings, not lookups. Adding ARB-driven translation is straightforward — install `gen_l10n` config and replace strings with `AppLocalizations.of(context).signIn` — but it's a separate, mechanical pass and we don't yet have translators.

## Supported locales

Defined in `locale_controller.dart`:

```dart
const supportedLocales = <Locale>[
  Locale('en'),
  Locale('ar'),
  Locale('fr'),
];
```

Add more by appending to the list and bumping the topbar's `_languageLabel` switch.

## How to enable real translation

1. Add to `pubspec.yaml`:

   ```yaml
   flutter:
     generate: true
   ```

2. Add `l10n.yaml` at the project root:

   ```yaml
   arb-dir: lib/l10n
   template-arb-file: app_en.arb
   output-localization-file: app_localizations.dart
   ```

3. Create `lib/l10n/app_en.arb` with the source strings, plus `app_ar.arb` and `app_fr.arb` translations.

4. `flutter pub get` regenerates `AppLocalizations`.

5. Add `AppLocalizations.delegate` to `localizationsDelegates` in `app.dart`, then replace literal strings with `AppLocalizations.of(context).<key>`.

The infrastructure to make all of that work end-to-end (controller, storage, switcher) is already in place from this phase.

## RTL caveats

A few places in the codebase use raw `EdgeInsets.fromLTRB(...)` or `Alignment.topLeft`. Most of those work fine because the visual position is intentional regardless of language (e.g. icon on the *start* side of a row). When you spot one that should flip, replace it:

- `EdgeInsets.fromLTRB(...)` → `EdgeInsetsDirectional.fromSTEB(...)`
- `Alignment.topLeft` → `AlignmentDirectional.topStart`

The dashboard sidebar / topbar are already directional-aware; the page builder + the auto-generated custom-entity forms inherit Flutter's defaults and flip correctly.

## Switcher UX

The topbar shows a small `🌐` (translate) icon plus the current locale code (`EN` / `AR` / `FR`). Clicking opens a popup menu of supported locales with a checkmark next to the active one. Selection is immediate and persistent.
